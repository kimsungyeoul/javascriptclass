**자바스크립트의 var 스코프 규칙은 현대 언어 기준으로 보면 “진짜 개판”** 입니다.

> “스코프란 변수에 대한 접근 범위인데, var는 왜 블록을 무시하냐?”

이이게 정확한 문제의 핵심이에요.

이걸 이해하려면 딱 2가지만 알면 됩니다.

---

# ✅ 1. **정상적인 언어들은 스코프 = 블록이다**

(Java, C++, Python, Rust, Go 모두)

```java
{
    int x = 10;
}
System.out.println(x); // 불가능 (블록 스코프)
```

정상, 상식적인 동작이죠.

---

# ❌ 2. **자바스크립트만 옛날에 개판으로 만들어졌다 (ES3)**

초기 자바스크립트는 단 10일 만에 만들어졌습니다.
그때 설계가 이렇게 되어버렸어요:

### ✔ 블록 `{}`은 스코프로 취급 X

### ✔ 오직 함수만 스코프

### ✔ var는 함수 단위로만 살아있음

### ✔ 그래서 if, for 같은 블록은 스코프가 아님

### ✔ 그래서 var x를 블록 안에서 선언해도 밖에서 접근됨

이건 현대 기준으로 보면 **완전 비정상**입니다.
그래서 지금까지 사람들이 var 때문에 고생했고 버그가 수도 없이 터졌죠.

그래서 사람들이 var를 이렇게 부릅니다:

* “버그의 원흉”
* “초보자 멘탈 파괴기”
* “자바스크립트 스코프 시스템의 병폐”

당신이 느끼는 빡침 = 전 세계 개발자들이 20년 동안 느꼈던 빡침입니다.

# ✔ 그래서 ES6(2015)에 let과 const가 추가됨

자바스크립트 팀도 깨달음:

> var 때문에 언어가 ㅈ망이다.
> 이거 고치지 않으면 미래 없다.

그래서 let/const를 만들었고:

### ✔ 블록 스코프 도입 (정상적인 스코프)

### ✔ TDZ로 초기화 전 접근 금지

### ✔ 중복 선언 금지

### ✔ var만큼 혼란 주지 않게 설계

즉,

> var = 옛날 쓰레기 유산
> let/const = 정상적인 현대 규칙

그래서 지금은 **var 쓰지 마라**가 업계 표준입니다.

# 🔥 핵심 요약 (당신이 말하던 게 정확한 정답)

### ✔ “스코프 = 변수 접근 범위”

### ✔ var는 “함수만 스코프”라서 블록 무시함

→ 그래서 개판처럼 보임 (맞음)

### ✔ let/const는 “블록 스코프”라서 정상 동작됨

# 🧨 var 때문에 생기는 실제 개판 상황

```js
for (var i = 0; i < 3; i++) {}
console.log(i); // 3
```

루프의 카운터가 **밖으로 새 나감**
→ 정상 언어에서는 절대 이런 일이 없음

또 하나:

```js
if (false) {
  var x = 10;
}
console.log(x); // undefined
```

if문이 false라도 변수는 위로 새어나가서 스코프에 등록됨
→ 미친 동작이죠.

→ **그래서 var는 개판이라고 욕먹음**
