# 🚨 **var가 너무 개떡같아서 → ES6에서 let과 const를 새로 만든 게 맞다.**

이건 감정적인 표현이 아니라 **실제로 TC39(자바스크립트 표준위원회)**가 그렇게 인정하고 문서를 만들었어요.

---

# 🔥 왜 var를 버리기 위해 let이 만들어졌는가?

## 1) var의 가장 큰 문제: **블록 스코프 없음**

```js
if (true) {
  var x = 10;
}
console.log(x); // 10 (블록 새 나감)
```

이거 때문에 전 세계 개발자들이 20년 동안 욕했습니다.
정상 언어였다면 당연히 에러죠.

---

## 2) var는 **호이스팅 + undefined 초기화** 때문에 버그 유발

```js
console.log(a); // undefined
var a = 10;
```

의도치 않게 값이 undefined로 나오며 버그를 숨깁니다.
실무에서는 "몰래 죽어있는 변수", "고스트 변수"라고 부르는 문제.

---

## 3) var는 중복 선언 허용 → 더 큰 버그

```js
var a = 10;
var a = 20; // 아무 문제 없음
```

이건 정말 말도 안 되는 언어 설계입니다.

---

## 4) var는 함수 스코프라서 for 루프도 개판됨

```js
for (var i = 0; i < 3; i++) {}
console.log(i); // 3
```

루프 변수(i)가 밖으로 빠져나오는 언어는 **자바스크립트밖에 없습니다.**

---

# 🔥 그래서 ES6(2015년)에서 공식적으로 결론을 냈다

### ✔ var는 버그를 너무 많이 만든다

### ✔ var는 블록 스코프를 무시해서 혼란을 만든다

### ✔ var는 TDZ 없이 undefined로 초기화돼서 안전하지 않다

### ✔ var는 한 스코프에서 중복 선언이 허용돼 위험하다

→ **그래서 let과 const를 새로 만들었다.**

당시 TC39 문서에도 “새로운 변수 선언 방식이 필요하다”라고 명시되어 있음.

---

# 🔥 let의 목적은 딱 3개

### 1) **블록 스코프 제공 → 정상 언어화**

(Java, C, Python 등과 동일한 스코프 규칙)

### 2) **TDZ 도입 → 선언 전 사용 금지**

(자바처럼 안전하게 설계)

### 3) **중복 선언 방지**

버그 예방

---

# 🔥 const는 let에 “재할당 불가” 기능만 얹은 것

(Java의 final과 유사)
